# V1.1: Soundness Audit 결과 및 수정 계획

> V1 스펙 기반 구현체에 대한 모듈별 soundness audit 결과.
> 감사 대상: 13개 크레이트 전체 소스 코드.

---

## 핵심 문제: 정수 산술 ↔ GF(2^128) 필드 산술 불일치

V1 설계에서 가장 근본적인 soundness 결함은 **회로 제약이 GF(2^128) 필드 산술으로 작성되어 있는데, EVM은 정수(ℤ/2^256) 산술을 요구**한다는 것이다.

GF(2^128)에서:
- 덧셈 = XOR (carry 없음)
- 곱셈 = carry-less multiply (정수 곱셈과 다름)

따라서 bitwise 연산(AND, XOR, NOT)에는 올바르지만, **정수 산술(ADD, MUL, DIV)의 제약은 의미적으로 불일치**한다.

---

## 발견 사항 — CRITICAL

### C-1. RangeCheck 회로 제약이 패스스루

**위치**: `circuit/src/constraint.rs` — tag 15

```rust
15 => Ok(out + in0),   // out = in0 — 범위 검증 없음
```

VM 레벨(`exec.rs`)에서는 `vr < 2^bits`를 검사하지만, **회로 레벨에서 비트 분해를 전혀 강제하지 않는다**. 악의적 프로버가 범위를 초과하는 값으로 Row를 생성해도 제약을 통과한다.

**영향**: `CheckDiv`의 `rem < divisor` 검증이 RangeCheck에 의존 → DIV/MOD 결과를 임의로 위조 가능.

**수정 방향**: 비트 분해 기반 range constraint 구현. `bits`개의 advice bit를 제공받아 `Σ 2^i · bit_i = value` 및 `bit_i ∈ {0,1}` (즉 `bit_i · (bit_i + 1) = 0`)를 제약.

---

### C-2. Mul128 제약 — GF(2^128) 곱셈 ≠ 정수 곱셈

**위치**: `circuit/src/constraint.rs` — tag 1

```rust
1 => {
    let shift128 = GF2_128::new(0x87, 0);   // x^128 mod p(x)
    Ok(in0 * in1 + flags * shift128 + out)   // GF(2^128) 곱셈
}
```

VM(`exec.rs`)은 `widening_mul_u128` (정수 곱셈)을 수행. `5 × 3 = 15`는 작은 값에서 우연히 일치하지만, 상위 비트가 개입하면 carry-less multiply와 integer multiply의 결과가 다르다.

**영향**: 모든 EVM MUL/MULMOD 연산의 soundness 붕괴. 정직한 프로버도 증명 실패하거나, 악의적 프로버가 GF(2^128) 곱셈을 만족하는 다른 값으로 위조 가능.

**수정 방향**:
- 방안 A: u128 정수 곱셈을 carry-chain 기반 bitwise 제약으로 재구현 (비용 증가)
- 방안 B: V1 원래 설계대로 32-bit micro-op으로 분해 (`Mul32` → Karatsuba), 128-bit 곱셈을 8개의 32-bit 제약으로 구성
- 방안 C: Lookup argument 도입으로 정수 산술을 테이블 기반 검증

---

### C-3. Add128 — carry chain 미검증

**위치**: `circuit/src/constraint.rs` — tag 0

```rust
0 => Ok(out + in0 + in1 + flags),   // out ⊕ in0 ⊕ in1 ⊕ flags = 0
```

GF(2^128) 덧셈은 XOR이다. `flags`가 carry correction을 인코딩한다고 문서에 있지만, **`flags` 자체에 대한 제약이 전혀 없다**. 프로버가 `flags = out ⊕ in0 ⊕ in1`로 설정하면 임의의 `out`이 통과한다.

**영향**: `1 + 1 = 999`도 유효한 증명으로 만들 수 있음. Rot128/Shr128/Shl128 (tag 5,6,7)도 동일한 문제 — `flags`에 대한 독립 제약 없음.

**수정 방향**: V1 원래 설계의 `Add32` carry chain을 구현. 128-bit 연산을 4개의 32-bit ripple-carry 가산기로 분해하고, 각 단계에서 carry bit ∈ {0,1}을 강제.

---

### C-4. CheckDiv — 필드/정수 불일치 + range check 부재

**위치**: `circuit/src/constraint.rs` — tag 12

```rust
12 => Ok(in2 + _advice * in0 + in1),   // GF(2^128): dividend = divisor · quot + rem
```

1. GF(2^128) 곱셈/덧셈이 정수 산술이 아님 (C-2와 동일)
2. `rem < divisor` 범위 검사가 회로 수준에서 없음 (C-1 의존)

**수정 방향**: C-1, C-2 수정 후 자동 해결. 정수 곱셈 제약 + 비트 분해 range check 필요.

## 발견 사항 — HIGH

### H-1. Boundary MLE가 PCS로 commit되지 않음

**위치**: `stark/src/prover.rs` (prepare 함수)

`boundary_mle`를 계산하고 `.sum()`을 구하지만 PCS commit에 포함하지 않는다. Proof에는 `boundary_constraint_sum` 값만 포함되고, verifier는 이 값이 0인지만 확인한다.

```rust
// prover.rs — boundary_mle는 PCS에 들어가지 않음
let boundary_mle = boundary_table.constraint_mle(beta);
let boundary_constraint_sum = boundary_mle.sum();
// ... pcs::commit에는 blinded_mle만 들어감
```

프로버가 `boundary_constraint_sum = 0`을 직접 설정하면 Seq boundary의 상태 불일치를 은폐할 수 있다.

**수정 방향**: Boundary MLE도 BatchTensorPCS에 추가 entry로 commit하고, 별도 sumcheck + PCS open으로 검증.

---

### H-2. jumpdest_hash — 32비트 충돌 위험

**위치**: `circuit/src/state_constraint.rs` — `jumpdest_hash()`

```rust
fn jumpdest_hash(table: &BTreeSet<u32>) -> u32 {
    // Blake3 → truncate to 32 bits
}
```

생일 공격으로 ~2^16 건의 서로 다른 jumpdest 테이블에서 충돌 발생 가능.

**수정 방향**: 해시를 GF(2^128) 원소(128비트)로 확장. Boundary row의 해당 컬럼을 u32에서 GF2_128으로 변경.

---

### H-3. constraint_sum ↔ shard total_sum 바인딩 불명확

**위치**: `stark/src/verifier.rs`

Verifier는 `proof.constraint_sum.is_zero()`와 `proof.recursive_proof.root_claim == proof.shard_batch.total_sum`을 각각 검사하지만, `constraint_sum`이 `shard_batch.total_sum`과 동일한 값인지 명시적으로 확인하지 않는다.

정직한 프로버의 경우 이 둘은 같지만, 악의적 프로버가 `constraint_sum = 0`으로 설정하고 `shard_batch.total_sum`은 다른 값으로 설정할 여지가 있다.

**수정 방향**: `assert!(proof.constraint_sum == proof.shard_batch.total_sum)` 검사를 verifier에 추가.

---

### H-4. Memory/Storage 연산에 ZK 제약 없음

**위치**: `circuit/src/constraint.rs` — tags 17, 23, 24, 26, 28

Store, MStore, MStore8, SStore, TStore 모두 `GF2_128::zero()` 반환 (제약 없음). Load 계열(16, 22, 25, 27)은 `out + advice = 0`만 확인 — advice가 올바른 메모리 값인지는 검증하지 않음.

문서에서 "memory consistency argument at a higher level"로 처리한다고 하지만, 현재 `consistency.rs`의 검증이 **비-ZK** (ProofTree 원문을 직접 검사)이므로 succinctness를 위반한다.

**수정 방향**:
- 방안 A: Permutation argument (LogUp 등)로 memory read/write 일관성을 회로 내에서 증명
- 방안 B: Sorted memory trace + adjacent-row transition constraint

## 발견 사항 — MEDIUM

### M-1. PCS 쿼리 중복 허용

**위치**: `pcs/src/lib.rs` — open/verify

```rust
let queries: Vec<usize> = (0..params.n_queries)
    .map(|_| (transcript.squeeze_challenge().lo as usize) % n_enc)
    .collect();
```

40개 쿼리 중 중복이 있으면 실효 보안 비트가 감소한다. `n_enc = 4k`가 2의 거듭제곱이므로 modular bias는 없지만, 중복은 정보량 손실.

**수정 방향**: 중복 제거 후 추가 squeeze로 보충, 또는 Fisher-Yates 샘플링.

---

### M-2. derive_open_point의 truncate/pad

**위치**: `stark/src/prover.rs`

```rust
point.truncate(total_vars);
point.resize(total_vars, GF2_128::zero());
```

`high_challenges.len() + low_challenges.len() ≠ total_vars`이면 soundness 손실:
- Truncate: 고차 변수의 challenge 소실
- Pad: 0이 아닌 challenge가 있어야 할 위치에 0

**수정 방향**: `assert_eq!(high_challenges.len() + low_challenges.len(), total_vars)` 로 엄격하게 검사. fan_in은 반드시 2의 거듭제곱 사용.

---

### M-3. CheckInv/CheckMul — 정수/필드 불일치

**위치**: `circuit/src/constraint.rs` — tags 13, 14

C-2와 동일한 근본 원인. GF(2^128) 곱셈과 정수 곱셈의 차이.

---

### M-4. Blinding이 constraint MLE에만 적용

**위치**: `stark/src/prover.rs`

Boundary MLE에는 blinding이 적용되지 않는다. H-1 수정 시 boundary MLE도 PCS commit에 포함되면 blinding 필수.

## 발견 사항 — LOW

### L-1. Consistency check의 succinctness 위반

**위치**: `stark/src/verifier.rs`

```rust
let errors = evm_types::consistency_check_all(tree);
```

Verifier가 전체 EVM 상태(스택 값, 메모리 내용)를 직접 검사한다. 증명 크기가 실행 크기에 비례하여 succinct proof 아님.

**수정 방향**: 장기적으로 consistency check를 회로 내로 이전. 단기적으로는 verifier가 ProofTree를 받는 것을 허용하되, 최종 프로덕션에서는 제거 대상.

---

### L-2. Additive FFT 코드의 MDS 최소거리 — formal proof 부재

**위치**: `pcs/src/code.rs`

`[4k, k, 3k+1]` MDS를 주장하지만 k=8 테스트 1건뿐. Novel polynomial basis 위 additive FFT code의 최소거리에 대한 formal verification이나 다양한 k 값 테스트가 필요.

**수정 방향**: 다양한 k 값에 대한 exhaustive weight 테스트 추가. k ≤ 16에서는 brute-force 가능.

---

### L-3. 미사용 batched_constraint 함수에서 beta 무시

**위치**: `circuit/src/constraint.rs` — `batched_constraint()`

```rust
let _ = beta;   // beta 파라미터 무시, cols[1]로 대체
```

실제로는 `batched_constraint_with_tag`가 사용되므로 현재 영향 없음. 혼란 방지를 위해 제거 또는 수정.

---

### L-4. Branch not_taken 경로의 boundary 처리

**위치**: `circuit/src/state_constraint.rs`

`rightmost_post(Branch)`가 `taken`만 반환. `not_taken` 경로의 post-state가 상위 Seq boundary에 올바르게 반영되는지 재확인 필요.

## 수정 우선순위 및 작업 계획

### Phase 1: 정수 산술 제약 재구현 (C-1, C-2, C-3, C-4, M-3)

V1 원래 설계대로 **32-bit micro-op 기반 제약 시스템**으로 복귀. 현재 128-bit 연산을 32-bit 단위로 분해:

```
현재 (128-bit, 잘못된 GF(2^128) 제약):
  Add128: out ⊕ in0 ⊕ in1 ⊕ flags = 0   ← carry chain 없음

수정 후 (32-bit, 올바른 integer 제약):
  Add32 × 4: out_i = in0_i + in1_i + carry_{i-1}
             carry_i = (in0_i · in1_i) | (carry_{i-1} · (in0_i ⊕ in1_i))
             carry_i ∈ {0,1}
```

**작업 목록**:
